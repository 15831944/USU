Application design tips Lecture 5100.05
  1. Common design mistakes
    1. Ganging pushbuttons into a large two-dimensional wall. Pushbuttons should be partitioned into smaller, manageable categories.
    2. Presenting a long list of radio buttons which requires scrolling. An option menu is preferable.
    3. Labeling menu selections as sentences. Generally, a single key word is possible and preferable. Make the key word as short, common, and precise as possible.
    4. Using a variety of font sizes and styles. Besides making a gaudy looking interface, this make uniform sizing of buttons and labels difficult.
    5. Using GUI objects for non-standard functions. This is confusing to the user. There is a sufficient breadth of function among existing objects that you can do everything you want with them.
    6. Using multiline menubars. Not only does it look awkward, but pulling down an upper menu obscures the lower choices. A minimum width for the application should be hardcoded, so that the menubar doesn’t shrink to the point that it requires two lines. Toolbars suffer from the same problems, but at least there are no obscuring menus.
    7. Using multiple words for a single menubar submenu. This is confusing to the user, who may think that two separate menus exist.
    8. Permitting objects to be clipped when the application is resized, or when individual portions of it are resized.
  2. Desirable design attributes
    1. The primary goal is to provide the user with an interface which fosters:
      1. Ease of use
      2. Ease of understanding
      3. Ease of recollection
      4. Avoidance of user errors
      5. Minimum user effort
      6. Maximum user flexibility
    2. Color, dimensions, graphics, and font choices should be aesthetically pleasing. Consider using someone with design experience to help with this.
    3. As part of ease of understanding, we should make the interface as intuitive as possible. This usually means that organization, graphics, and actions match the user's real life experiences as closely as possible. We should avoid importing metaphors from other application domains, which have no meaning in this application domain. An example would be using a gun icon to delete a file in a text editor.
    4. A good hierarchical structure for menus and functions will allow the user to learn comfortably. Start with reasonable defaults, and allow special cases at the first level, fancier options at the second level, expert tailoring at the third level, etc.
    5. Pay particular attention to visual placement of objects.
      1. Menubar at the top
      2. Divide the work region into panes if necessary
      3. Make each pane scrolled if necessary
      4. Put the toolbar at the bottom (OSF look and feel) or immediately below the menubar (Windows L&F)
  3. As a part of maximum user flexibility, the user should always be presented with as wide a range of choices as possible. That is to say, the application should be as modeless as possible.
