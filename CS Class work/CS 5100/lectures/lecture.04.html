GUI Design Principles Lecture 5100.04
  1. Steps
    1. Define the problem the customer wants solved.
    2. Conduct user profile analysis so as to model the user portion of the system.
    3. Conduct task analysis to determine atomic actions and scenarios for accomplishing tasks.
    4. Define the computer objects and functions that correspond to the task domain.
    5. Design the appearance and behavior of the user interface. Create a prototype.
    6. Evaluate the design by using the prototype.
    7. Modify the analysis and/or design, create a new prototype, and repeat until satisfied. There may be hundreds of prototypes before we are satisfied.
    8. Typically, there are two approaches to appearance design.
      1. The first is bottom-up, in which we first choose the individual features that we want to appear. We then try to integrate these into a coherent interface. This tends to be susceptible to "featuritis", in which the user becomes overwhelmed by an incoherent mass if individual actions.
      2. The second is top-down, in which we choose a unifying metaphor, then add such features as fit into the metaphor. The problem here is that the metaphor must be very carefully chosen, or we will either lose features or fracture the metaphor trying to fit them in.
  2. Evaluation must be based on actual experience of typical users in typical circumstances performing typical tasks.
    1. "think aloud" evaluation involves asking the user to vocalize all thoughts while using the software. This gives hints on difficult, confusing and misleading aspects of the interface.
    2. User instrumentation involves using video cameras, eye tracking devices, etc. To determine exactly what actions are performed by the user. This can be used in conjunction with software instrumentation, in which all input and output events are recorded. Instrumentation tries to record sequences of atomic actions, the amount of time needed to accomplish each action, the frequency with which one action follows another, the number of errors, and what seemed to cause them, where the user looks on the screen, whether mouse or key interaction is used, etc.
    3. Ethnographic studies observe users in action without interfering.
    4. This information can be used in many ways. Some of them are:
      1. Which windows or buttons should be put next to each other to minimize mouse action, since these are usually used one after the other (principle of locality);
      2. Which actions should be put at the top of the menu hierarchy because they are frequently used, which are unusual and should be put at the bottom;
      3. Which features generate an unusual number of errors ; this is sometimes because the feature doesn't represent the user's concept of what's going one (semantic distance);
      4. Which features require too much effort;
      5. Which atomic actions are never used.
  3. Specific elements to consider in any interface:
    1. Macro facilities to turn common scenarios into a single atomic action;
    2. Redo/undo facility;
    3. Cut/copy/paste facility with a system-wide clipboard;
    4. On-line documents, help, wizards, and coaching;
    5. Default behavior and appearance to satisfy either the naive users or the largest group of users;
    6. Customization for those who are not satisfied with defaults.
