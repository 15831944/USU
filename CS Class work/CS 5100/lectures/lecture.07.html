Introducing JFC Lecture 5100.07
  1. The Java Foundation Classes (JFC) are a set of Application Program Interfaces (APIs) which are used to build applications. They are based on Swing architecture.
    1. JFC APIs include Java 2D graphics and Imaging, Accessibility, Drag-and-Drop, Input Method Framework, and Swing
    2. Swing (as implemented extending the AWT) is really a framework for the construction of GUI components
    3. Swing characteristics include
      1. Use of lightweight components
      2. Pluggable look-and-feel
      3. A new (delegation) event model
      4. Support for localization and accessibility and beans
      5. AWT compatibility
      6. Support for MVC architecture
  2. The Abstract Windowing Toolkit (AWT)
    1. AWT is the primary Java GUI toolkit. Its features are implemented in packages, like java.awt.color, java.awt.image, etc., as well as java.applet and java.beans.*;
    2. Beans are not technically part of AWT, but are a component architecture aimed at GUI elements.
    3. AWT depends on native peers (elements of window systems on platform hardware) to actually display screen objects. Thus, their appearance has the native look-and-feel.
    4. Swing's Pluggable Look-and-Feel API provides a characteristic Java look-and-feel independent of the platform. These are supported by javax.swing.plaf.
    5. Heavyweight components have native peers, and use a lot of memory. Lightweight components are drawn directly on a native peer, and use far fewer resources.
  3. AWT components are listed in Table 2.2.
    1. All Java classes are derived from Object. All visible components are derived from Component. Each AWT component has:
      1. A Graphics object
      2. Foreground and background colors
      3. Minimum, maximum, and preferred sizes
      4. Dimension and screen location
      5. Font and Cursor
      6. Locale, parent, and peer
    2. Containers are a special subclass which can manage and draw other components within their own windows. Container subclasses include Applet, Dialog, Frame, Panel, ScrollPane, and Window. Each Container has an associated layout manager, some of which are Border, Grid, Flow, Card, and GridBag.
    3. AWT components frequently have Swing analogues (e.g., Frame and JFrame). However, there are Swing components with no AWT analogue, like JTable.
    4. Component and Container are both abstract classes. No objects of these types can be instantiated.
    5. The Swing JComponent is derived from java.awt.Container, so all Swing components are in reality containers.
  4. The Delegation Event Model
    1. Program events are generated to represent external asynchronous actions, like mouse and key actions.
    2. The object in which the event occurs is the source. An event listener (handler) of a particular category is added to the object, and is executed when an action of that category happens.
    3. Adding a listener to a component is often called "registering" it.
  5. The JFC Containment Model
    1. In AWT, all containers but Canvas are heavyweight, and nested containers manage each other, at great cost.
    2. In JFC, most containers are lightweight, and really only the top-level container adds children. However, it delegates layout responsibilities to the other lightweight containers within it.
    3. JApplet, JDialog, JFrame, JInternalFrame, and JWindow use the JFC Containment Model. This means that each has some standard containers within it:
      1. root pane manages glass, content, and menubar panes
      2. content pane is the basic object window; all components are added to it
      3. layered pane manages the content pane to allow overlays and stacking as well as general layout management
      4. glass pane can intercept events before they go to other nested components, and facilitate drawing uniformly over several nested components.
      5. menubar provides menus
      6. Some Containers are missing some of these.
    4. Remember that the containment hierarchy has nothing to do with the inheritance hierarchy.
  6. The MVC model
    1. Models are internal data structures that hold data
    2. Views are externally visible graphical objects that display data
    3. Controllers are responsible for communicating values to models and their corresponding views, trapping user events and sending update messages.
    4. Typically in JFC, views and controllers are collapsed into a single object. In fact, most classes are defined so they can perform as any one of the three.
    5. Some examples of JFC Abstraction Models include BoundedRangeModel, ButtonModel, and ListModel. These are interfaces.
      1. They have default implementations, like DefaultBoundedRangeModel.
      2. For instance, an implementation of ListModel must provide the methods getElementAt and getSize. They support the Model
      3. An implementation must also provide code for AddListDataListener and RemoveListDataListener. They support the Controller.
