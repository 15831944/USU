AWT and Swing Event Specifics Lecture 5100.20

  1. AWTEvents
    1. The AWT events include both container and component events. Since handling is very similar, even though the semantics are different, they are treated together here.
    2. It becomes very important in building an application to understand what events might be generated by the objects. While there is too much to memorize, it is good to try to get a general idea.
  2. Component events are generic for all components. Specific components will generate additional events under specific conditions.
    1. Constants: Event subtypes are shown, hidden, resized, added.
    2. Method: getComponent identifies which component generated the event.
  3. Container describes interactions between containers and their child components. Container events are generic for all containers.
    1. Constants: There are event subtype constants indicating child added and child removed
    2. Methods: getContainer (where it happened) and getChild (which one caused it)
  4. Focus deals with change in keyboard focus at the component level. Changes can be temporary, due to deactivation or scrolling or mouse exit, or permanent, due to program or user generated actions. Temporary changes will be undone when scrolling stops or the window is reactivated.
    1. Only one component can have focus at a time. Focus can be changed by mouse clicks or program action, or traversed by the tab key or program action.
    2. There is frequently a visual signal (shading, highlighting, etc. ) of the component with focus.
    3. Constants: There are constants indicated that focus was gained or lost
    4. Method: isTemporary
  5. Input events are an abstract class and embrace both key and mouse events.
    1. Constants: Masks for modifier keys and mouse buttons
    2. Methods:
      1. consume, isConsumed
      2. is[modifier]Down tells whether each of the four modifier keys was down when the event occurred. isShiftDown, for instance, tells whether the user wanted to indicate an upper case or lower case letter.
      3. getWhen gets the time of the event
      4. getModifiers gets a combined mask for all modifiers. These include alt, shift, control, meta, and the three mouse buttons.
  6. Key events deal with user keyboard actions
    1. Constants
      1. There are different Virtual Key constants, like VK_LEFT, which is associated with the left cursor key or VK_F3 for the function 3 key. Some keys are "action" keys, like the cursor control, function, and special keypad keys.
      2. There are constants indicating three event subtypes: key pressed, key released, and key typed. Typing only occurs for keys that represent characters, that is, non-action keys.
    2. Methods
      1. getKeyChar returns the character associated with a typeable, non-action key.
      2. getKeyCode returns the virtual key code for the key used. This method does not work for KEY_TYPED events.
      3. GetKeyModifiersText returns a string ("Alt+Shift") describing which modifier keys were down.
      4. GetKeyText returns a string ("Home") describing which key was pressed.
      5. There are methods to set the code, char, and modifiers. These are useful when we want to do things like automatically show all characters in uppercase, or show password characters as asterisks.
      6. The isAction method tells whether the key pressed was an action key.
      7. Since most components are not interested in keystrokes, AWT doesn’t even bother to deliver key events to them.
  7. Mouse events deal with mouse clicks and movements
    1. Constants
      1. Masks for mouse pressed, released, and clicked
      2. Masks for mouse moved and dragged
      3. Masks to indicate a mouse entered, or exited, the bounding rectangle of an object
    2. Methods
      1. getX and getY indicated the (relative) coordinates at which the mouse event occurred. getPoint returns the same information as a point.
      2. isPopupTrigger indicates whether the mouse action should pop up the menu associated with the object where the event occurred
      3. getClickCount tells how many clicks occurred within the double-click period
    3. There is no separate MouseMotionEvent, even though there are corresponding Listeners and Adapters, each of which is specialized to handling certain kinds of the above events. In Swing, there is a MouseInputListener interface which does both, but no corresponding add method.
  8. Paint events include anything, like exposure, which causes a component to be repainted. Application programs don’t handle paint events directly. This is done internally by AWT.
  9. Action represents clicking a button or hitting the enter key when they trigger an action
      1. Constants: there are masks indicating which modifier keys (shift, alt, control, meta) were in effect at the time of the action
      2. Methods
        1. getModifiers
        2. getActionCommand returns the logical, language-independent action command associated with this event
  10. Adjustment deals with scrollbar events
    1. Constants: There are constants indicating unit down or up, block down or up, or block drag
    2. Methods
      1. getAdjustable returns the adjustable object in which the event occurred
      2. getValue returns the current value (scrollbar scale setting)
      3. getAdjustmentType returns one of the constants, indicating the kind of event
  11. Item events relate to the selection of an item in a list. Item events are semantic events which are generated by objects which implement the ItemSelectable interface. Besides adding and removing listeners the interface dictates a method to getSelectedObjects. Checkbox, CheckboxMenuItem, Choice and List are all selectable.
    1. Constants: there are constants indicating that an item’s state was changed, and that the state change was either a selection or deselection
    2. Method
      1. getItemSelectable indicates the list in which the event occurred
      2. getItem returns an object, usually a string, which describes the selected item
      3. getStateChange, returns either the selection or deselection constant
  12. Window events are those that are associated with the appearance and disappearance of windows on the desktop.
    1. Constants
      1. Masks for iconified and deiconified, that is the collapse of a window into an icon on the taskbar or desktop, or its restoration
      2. Masks for activated and deactivated, that is bringing the window to the front, or replacing it at the front with another window
      3. Masks for window opened, closing, and closed. Closing refers to events in which the user requests the window be closed, closed refers to the ultimate closure of the window. Closing is an appropriate time to dispose of window resources.
    2. getWindow returns the window which generated the event.
  13. Text events are very simple. There is a TEXT_VALUE_CHANGED constant and a paramString method. There is a much more complex text package in Swing.
  14. Custom Events can be created using id numbers above the last id number reserved for the system. This can happen when we create our own components, or when we want to communicate between threads.
  15. Swing Events
    1. These tend to be semantic rather than component events. Remember that Swing components can still generate and handle many AWT events.
    2. There are also four Swing events which are actually in the AWT package hierarchy.
    3. Ancestor events deal with occasions when something happens to a component’s ancestor.
      1. Subevent type constants for ancestor added, removed, or moved.
      2. Methods to get the component generating the event, the ancestor of that component which was added, removed, or moved, and the parent of the ancestor.
    4. Caret events deal with the caret position in text, which is sometimes different from the cursor position. For instance, if we are carrying out a selection, the original position is the "mark" and the current position is the "dot". The caret represents the position at which an input would occur. The two methods get the mark and the dot.
    5. Change events occur when an object changes state. Thus in the Swing hierarchy, it is used instead of a selection event for JCheckboxes. There are no constants and methods.
    6. Hyperlink events are generated in editor panes using the HTML editor kit when the cursor enters, leaves, or activates a hyperlink.
      1. Constants: Subevent ids for entered, exited, and activated.
      2. Methods: getDescription (a string for the URL), getURL, getType
    7. InternalFrame events mirror those for regular frames (activated, iconified, etc.) but apply to events that occur in internal frames. For instance, we might have a multiple document interface, say an editor with multiple documents open, and want to iconify the frame for a particular document and select another.
