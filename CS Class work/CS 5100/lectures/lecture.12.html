Swing Concepts Lecture 5100.12

  1. The Swing Tool Set
    1. Comprised of:
      1. A different architecture than the Abstract Windowing Toolkit. It is largely compatible with AWT, but extends it by having lightweight components with their own look and feel
      2. A new set of visual components, many of which mirror AWT components. For instance, JFrame or JButton replace Frame and Button.
      3. A set of non-visual components which support the abstractions added to AWT in the Swing architecture.
    2. The current standard is to have Swing components in the javax.swing package. This is javax, not java. That is, it is an extension of standard java, not standard java.
    3. While all Swing classes are derived from AWT classes, it doesn't work to mix AWT and Swing components in the same application, since the heavyweight AWT components will inevitably be rendered on top of the lightweight Swing components.
    4. It is the fact that JDK 1.1 AWT Container and Component classes are abstract, and have no native peers, that permits the creation of lightweight classes.
    5. In order to program with Swing, one must have JDK 1.1 and also have the Swing classes. These are most commonly found in a swing.zip file.
    6. The book also describes some features of Java2, which is JDK 1.2.
    7. The abstractions and features added by Swing include:
      1. Lightweight components and containers
      2. A containment model in which containers delegate responsibilities to specialized subcontainers.
      3. The inclusion of standard data models and implementations.
      4. Pluggable look-and-feel
      5. Support for the Model-View-Controller architecture.
      6. ToolTips
      7. Internationalization
      8. Attachment of action commands to all components.
  2. JComponent
    1. Since it is derived from Container, all Swing components can contain other components. Thus we can have images on buttons, rather than simple text labels. This allows for very sophisticated components.
    2. JComponent (and JContainer, for that matter) are not responsible for providing native peers. Rather, they provide methods to support the determination of appropriate native peers. Specific subclasses, if heavyweight, will implement those methods to choose appropriate peers.
    3. Since it is lightweight, classes derived from JComponent can also be lightweight.
    4. All classes derived from JComponent have the following properties:
      1. A data model (getModel, setModel)
      2. A PLAF (getUI, setUI, updateUI)
      3. A ToolTip (createToolTIp, getToolTIpLocation, getToolTipText, setToolTip)
      4. A root pane to provide a delegate for containment and management of children (getRootPane, getContentPane)
      5. A locale for internationalization (getLocale, setLocale)
      6. An action command
      7. Double buffering to minimize flicker when rendering. Double buffering means that an image is created in off-screen memory, then transferred very rapidly to the screen. (setDoubleBuffered)
      8. Accessibility (getAccessibileContext)
  3. Swing Containers
    1. Some are heavyweight, others are not.
    2. Have structured containment instead of the flat AWT containment
    3. Permit Z-ordering, thus allowing stacking of children, container-wide drawing and popups, etc.
    4. Swing containment displays good object-oriented design in the sense that:
      1. It factors different aspects of containment into different subcontainers, so that containers which don't need those aspects don't have to use them
      2. It avoids repetition, and promotes reuse
      3. It separates abstract containment issues from specific implementations
      4. It maintains compatibility with AWT
